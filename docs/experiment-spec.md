# Experiment Spec

This document locks the core experimental contract for the proof-of-concept testbed. The goal is to ensure the offline HDC search, the classical beta baseline, and the final closed-loop evaluation all use the same controller-visible signal path and the same fairness rules.

| Parameter name | Value | Design choice |
| --- | --- | --- |
| Controller-visible sampling rate | 250 Hz | Locked global rate for all offline and online experiments. High enough for beta-band structure, low enough to keep compute simple and fair across controllers. |
| Decision cadence | 50 ms | Locked global decision interval. Both controllers evaluate the latest rolling window on the same schedule. |
| ODE solver output handling | Variable-step ODE output is never passed directly to a controller | The simulator may integrate with `solve_ivp`, but controller logic must only see a fixed-rate stream. This avoids irregular sampling artifacts and keeps offline and online evaluation aligned. |
| ODE resampling method | Sample solver output onto a uniform 250 Hz time grid per simulation chunk | Use a fixed-rate resampled stream as the contract boundary between the simulator and the controllers. This is deterministic, easy to reproduce, and avoids leakage between implementation details and evaluation logic. |
| Controller-visible signal | One scalar LFP-like surrogate derived from a fixed population-average state variable across STN-GPe neurons | For the MVP, use a simple fixed population-average state variable rather than a more biophysical synaptic-current model. This is sufficient for comparing decision statistics as long as the same signal is used everywhere. |
| LFP surrogate policy | Keep one surrogate definition fixed across all experiments | The testbed is evaluating HDC versus beta-threshold control, not optimizing the observation model. The surrogate must not change between offline tuning and final evaluation. |
| Online filtering policy | Causal processing only for controller-visible signals | No non-causal filtering is allowed in any online controller path. This keeps the benchmark deployable in principle and prevents future leakage from the signal-processing pipeline. |
| Beta baseline input path | Same resampled surrogate stream used by HDC | The classical baseline and the HDC controller must start from the same signal. The only intended difference is the decision statistic. |
| Beta-band statistic | Causal beta-band power estimate from the shared surrogate stream | The classical controller uses a causal beta-band extraction and power estimate because beta power is its decision signal. |
| HDC input path | Raw resampled surrogate window with online-safe normalization | HDC should operate on the same shared signal stream, without privileged preprocessing that changes the comparison target. |
| Default window length | 512 ms | At 250 Hz this is 128 samples. It is long enough to capture beta structure while still being responsive enough for adaptive control. |
| Window stride | 50 ms | Rolling overlapping windows are advanced at the decision cadence so both controllers act on the same temporal context. |
| Default normalization | Per-window z-score | Chosen as the safe online default because it does not require carrying reference statistics and is naturally leakage-resistant. |
| Validator normalization policy | Per-window z-score only | Locked for MVP to minimize assumptions and reduce confounding in encoder comparison. |
| Stimulation burst duration | 200 ms | A short fixed burst is simple to analyze and matches the intended adaptive control pattern better than continuous stimulation. |
| Post-stimulation lockout | 200 ms | Prevents immediate retriggering, reduces chatter, and makes the controller state machine comparable across methods. |
| Controller hysteresis policy | Both adaptive controllers should use matched on/off logic where possible | The comparison should isolate beta power versus HDC margin, not differences in controller mechanics. |
| Controller parity rule | Same stream, same window, same stride, same epoch duration, same lockout, same hysteresis structure if possible | Baseline fairness depends on keeping the controller structure matched and changing only the decision statistic. |
| Dataset unit | Whole trajectory / run / seed | Data splitting must happen at the trajectory level rather than the window level so overlapping windows do not leak across splits. |
| Train/validation/test split rule | Split trajectories first, then extract windows within each split | This prevents inflated offline performance from overlapping-window leakage. |
| Default split proportions | 60% train / 20% validation / 20% test by trajectory count | Simple default that supports training, threshold calibration, and final held-out evaluation. |
| Healthy specificity holdout | Reserve long healthy-only trajectories for false-trigger testing | Good suppression alone is not enough; the adaptive controllers must also avoid unnecessary stimulation in healthy regimes. |
| Threshold calibration rule | Calibrate beta and HDC thresholds on validation trajectories only, after encoder ranking on validation is complete | Train data builds models; validation data is used for ranking and threshold selection; test data remains untouched until final evaluation. |
| Frozen-before-test rule | Freeze encoder family, readout choice, ranking rule, thresholds, smoothing, hysteresis, and lockout before test and final closed-loop runs | Prevents post hoc tuning and keeps the reported comparison credible. |
| Offline validator scope | Encoder search may vary dimension, bins, initialization, and readout under a fixed signal contract and is ranked on hold-out validation only | Window length is locked to 512 ms and normalization is locked to per-window z-score so validator comparisons focus on encoder/readout design without test leakage. |
| Encoder ranking rule | Primary: balanced accuracy on clean validation healthy vs pathological windows; Guardrail: minimum AUROC on onset and recovery validation windows; Tie-breaker: mean encoding time per window | Enforces one frozen selection rule, preserves transitional robustness, and prefers lower-latency encoders when primary scores are close. |
| Validator selection funnel | Run full offline validation search, keep top 2-3 validation-ranked candidates for small validation-only controller sweep, then freeze one final HDC configuration before touching test | Balances efficiency with rigor by avoiding closed-loop test leakage while still checking practical controller behavior before final lock-in. |
| Online evaluation metrics | Mean beta power, pathological occupancy, suppression latency, duty cycle, pulse count, mean decision time per window, and memory use | These metrics capture control quality, stimulation efficiency, and runtime cost. |
| Additional specificity metric | Healthy false-trigger rate and unnecessary stimulation epochs during healthy-only runs | This addresses the main specificity gap in the current plan. |
| Additional simulator-native metric | Include at least one secondary metric such as synchrony, burst fraction, or distance to healthy PSD profile | The final evaluation should not be judged only by the same beta biomarker used by the classical baseline. |
| Scope of this contract | Global for the proof-of-concept testbed | This specification applies across simulator validation, offline HDC search, and closed-loop controller evaluation unless a later revision explicitly changes it. |
